1 What this is — in plain words

You use Hyperspace (the SAP hai proxy on localhost:6655) to access Claude — same as Mac Terminal uses for Claude Code. But Hyperspace speaks Anthropic's Messages API, which most AI-powered IDEs don't natively understand. They speak Ollama or OpenAI formats instead.

The Hyperspace Bridge is a small Node.js translator running on your Mac. It listens on two local ports, pretends to be Ollama and OpenAI, and forwards every request to Hyperspace in the format it expects. Responses stream back in real time, token by token.

💡
Analogy: Think of it as a power adapter when travelling. Your laptop plug (the IDE) is one shape, the wall socket (Hyperspace) is another shape. The bridge is the adapter — nothing changes about the laptop or the wall.

Why this matters

Once running, ANY local AI tool that speaks Ollama or OpenAI auto-detects your Hyperspace setup as if it were a local Ollama server. No more cloud SaaS lock-in to Cursor/Copilot/etc. — your editor uses Claude through your own corporate proxy, with proper licensing and zero data leak risk.

Auto-detection of your Hai API key

The bridge looks for your ANTHROPIC_AUTH_TOKEN in this order:

  1. $HAI_API_KEY environment variable (if set)
  2. $ANTHROPIC_AUTH_TOKEN environment variable (if set)
  3. ~/.claude/settings.json → env.ANTHROPIC_AUTH_TOKEN (where hai configure claude-code writes it)

That's why it "just works" if your Mac is already set up for Claude Code in Terminal.

3 What is Continue.dev?

Continue (continue.dev) is an open-source AI coding assistant that runs as an extension inside your IDE — VS Code, Cursor, Antigravity, JetBrains, and others. Think of it as a chat panel + autocomplete + inline-edit tool that lives alongside your code.

The key thing that makes Continue special for our setup: you choose where its AI comes from. Unlike Cursor's built-in chat (locked to Cursor's cloud) or Antigravity's Agent (locked to Google's Cloud Code Assist), Continue lets you point it at any AI provider you want — including a local Ollama or OpenAI-compatible endpoint, which is exactly what the Hyperspace Bridge exposes.

What it does inside your IDE

FeatureHow it feels
Chat sidebarA panel on the left like ChatGPT, but it knows your codebase. Ask anything, paste code, get answers.
Inline edit (⌘I)Select code, describe a change, Continue rewrites it in place with a diff you accept or reject.
Tab autocompleteLike Copilot — suggestions appear inline as you type. Uses Claude Haiku via the bridge for speed.
Agent modeContinue can read files, run terminal commands, apply diffs across multiple files. You saw this in the screenshot where Claude offered a date shell command with an "Apply" button.
@-mentionsType @code, @docs, @terminal, @codebase to give Claude focused context from your project.

Why we use it for Antigravity and Cursor

Both Antigravity IDE and Cursor hard-wire their built-in AI to their own cloud — Antigravity to Google's daily-cloudcode-pa.googleapis.com, Cursor to cursor.sh. Neither IDE has a setting that says "use my own endpoint instead."

Continue.dev sidesteps this completely. It runs as a regular extension (no special privileges), brings its own UI and its own AI pipeline, and that pipeline IS configurable via ~/.continue/config.yaml. We point that file at http://localhost:11434 (the bridge), and now Continue → Bridge → Hai → Claude works end-to-end. The IDE's built-in AI stays untouched — you just ignore it.

🔌
The chain in one line: You press ⌘L in Antigravity → Continue extension activates → sends Ollama-format request to localhost:11434 → bridge translates to Anthropic format → forwards to Hai on :6655 → streams Claude's response back, token by token, to your IDE.

How Continue knows about Hyperspace

In your screenshot you saw 4 models in Continue's picker — Claude Sonnet (Hyperspace), Claude Haiku (Hyperspace), Claude Opus (Hyperspace), Claude Sonnet (OpenAI path). They all came from this config in ~/.continue/config.yaml:

models:
  - name: Claude Sonnet (Hyperspace)
    provider: ollama              # ← Continue speaks Ollama protocol
    model: claude-sonnet-latest
    apiBase: http://localhost:11434   # ← talks to YOUR bridge
    roles: [chat, edit, apply]

Continue thinks it's talking to a local Ollama server. The bridge is pretending to be Ollama. Hai is what actually returns Claude responses. Continue doesn't know or care — it just sees a working backend.

4 Continue styling patch — Cursor-native look

Out of the box, Continue.dev's UI uses its own font sizes, dimmed dropdown items, and depends on a CSS variable (--vscode-input-background) that Cursor's theme leaves unset or sets to a transparent value. The result: dropdown items looked faded, the model picker font was tiny, and a rainbow "currently generating" gradient that's meant to be a 1px border filled the entire user message bubble.

We patched Continue's bundled CSS to match Cursor's native chat sizing exactly. The patch is a CSS file appended to Continue's index.css in both Cursor and Antigravity. Pristine originals are saved as index.css.original so you can revert any time.

What the patch fixes

IssueRoot causeFix
Rainbow filled the whole bubble instead of being a 1px border Continue's inner editor uses background-color: var(--vscode-input-background, #2d2d2d). Cursor's theme sets that variable to a transparent value, so the gradient behind shows through. Force --vscode-input-background: #1e1e1e !important on html, body, and :root
Model dropdown items transparent (faded) Continue uses Tailwind opacity-50–opacity-90 on non-selected items Override all opacity-25 through opacity-95 utility classes to 1. Preserve opacity-0 (used for hidden checkmarks).
Mode dropdown ("Chat/Plan/Agent") overlapped "Last Session" Continue's floating popover had no opaque background — content behind bled through Opaque background-color + border + box-shadow on [data-floating-ui-portal], [role="listbox"], [role="menu"]
Toolbar row tinted gray (Agent / Claude Sonnet / Enter) Same --vscode-input-background mismatch from Cursor's theme Same global CSS variable override
Dropdown font too small (12px) Tailwind text-xs default Force font-size: 13px on dropdown rows — matches Cursor native picker exactly
Icons misaligned in dropdown rows SVGs inherited the bumped font-size and grew with text Pin SVGs to 14×14px with flex-shrink: 0

How we found the root cause

The rainbow-fill problem was the hardest one to track down. The path went like this:

  1. Continue's user-input editor is a 2-layer styled-component: an outer wrapper with padding: 1px + an animated gradient background, and an inner editor with an opaque background that masks the gradient — leaving only the 1px gap visible as a "border."
  2. The inner editor uses background-color: ${rv} where rv is computed by Continue's theme helper oi("input-background").
  3. That helper returns the literal string "var(--vscode-input-background, #2d2d2d)".
  4. In Antigravity: --vscode-input-background is a solid dark color, so the mask works → 1px border visible.
  5. In Cursor: the variable is set to something transparent at body level, so the mask becomes transparent → gradient fills the bubble.
  6. Fix: override the variable globally with !important at html/body/:root level. Continue's existing code does the rest.

The final patch (v8)

/* hyperspace-bridge-style-fix v8 — appended to Continue's index.css */

/* (1) Force the VSCode token Cursor leaves transparent. This is the
       single fix that makes the rainbow show as a 1px border, not fill. */
html, body, :root {
  --vscode-input-background: #1e1e1e !important;
  --vscode-menu-background: #1e1e1e !important;
  --vscode-menu-border: #454545 !important;
}

/* (2) Full opacity on dim utility classes (model picker items). */
.opacity-25, .opacity-30, ... , .opacity-90, .opacity-95 {
  opacity: 1 !important;
}

/* (3) Opaque dropdown popovers — Mode & Model pickers no longer bleed through. */
[data-floating-ui-portal], [role="listbox"], [role="menu"] {
  background-color: #1e1e1e !important;
  border: 1px solid #454545 !important;
  border-radius: 6px !important;
  box-shadow: 0 4px 14px rgba(0,0,0,0.5) !important;
}

/* (4) Cursor-native 13px dropdown rows + 14×14 icons. */
[role="listbox"] > li, [role="menu"] > div {
  font-size: 13px !important; line-height: 1.4 !important;
  padding: 5px 10px !important; min-height: 26px !important;
}
[role="listbox"] > li > svg, [role="menu"] > div > svg {
  width: 14px !important; height: 14px !important;
}

/* (5) Disabled items stay dimmed (correct UX signal). */
[aria-disabled="true"], [disabled] { opacity: 0.5 !important; }

6 Agent mode fix — YAML extension required

Continue's Agent mode silently stopped working without a specific VS Code extension installed. This was the hardest bug to diagnose — the bridge was healthy, the extension was loaded, but no chat requests ever reached the bridge.

Root cause

Continue requires redhat.vscode-yaml to register its config.yaml schema. Without it, Continue throws this error silently at startup:

Failed to register Continue config.yaml schema, most likely,
YAML extension is not installed
CodeExpectedError: Unable to write to User Settings because
yaml.schemas is not a registered configuration.

When this happens, Continue's entire model backend fails to initialize. Agent mode accepts input but sends nothing to the bridge. The bridge log shows only health checks, never chat requests.

The fix

install.sh step 7 now auto-installs both required extensions in every detected IDE:

ExtensionWhy needed
Continue.continueThe AI sidebar itself
redhat.vscode-yamlRequired for Continue to initialize its model backend. Without it, Agent mode silently fails.

Manual install (if needed)

# Cursor:
"/Applications/Cursor.app/Contents/Resources/app/bin/cursor" \
  --install-extension redhat.vscode-yaml

# Antigravity:
"/Applications/Antigravity IDE.app/Contents/Resources/app/bin/antigravity-ide" \
  --install-extension redhat.vscode-yaml
⚠️
Symptom to watch for: bridge log shows only GET /health → 200 with no POST /api/chat lines — Continue is running but not sending requests. Fix: install redhat.vscode-yaml and restart.

7 Hyperspace status indicator in the IDE status bar

A green "Hyperspace" text appears in the IDE status bar (bottom right, next to "Continue") whenever the bridge is reachable. No dot, no icon — just the word in green. When the bridge is offline, it disappears completely.

💚
Online:  ✓ Continue (NE)   Hyperspace
Offline:  ✓ Continue (NE)

How it works

We patched Continue's extension.js (the running extension, not the webview) to add a dedicated VS Code status bar item with vscode.ThemeColor("terminal.ansiGreen"). A polling loop pings http://127.0.0.1:11434/health every 15 seconds and shows/hides the item based on the response.

PropertyValue
Poll interval15 seconds
First checkImmediately on IDE startup
Cost~1ms loopback request every 15s — negligible
When offlineItem is hidden — no noise, no error state
Colorterminal.ansiGreen — VS Code's standard green token

9 IDE compatibility

Not every IDE lets you redirect AI traffic to a local endpoint. Here's the honest breakdown, with hard evidence from each IDE's source code.

Zed ✓ Native — works directly

Zed has first-class language_models.ollama and language_models.openai_compatible in its settings. Already configured at ~/.config/zed/settings.json.

Open Assistant panel → pick "Claude (via Hyperspace)" → chat. Done.

Antigravity IDE ⚠ Native AI uses Google gRPC — use Continue.dev sidebar instead

Why the native panel can't use the bridge:

  • Antigravity is a Google Labs IDE (built on Codeium's codebase)
  • Native "Agent" panel uses daily-cloudcode-pa.googleapis.com with gRPC-Connect over HTTP/2 and proprietary Codeium Protobuf schemas
  • Auth is Google OAuth 2.0 Gaia tokens — not API key based
  • No public schema, no Anthropic-compatible mode, no settings field for a custom endpoint

Solution that works: Continue.dev v2.0.0 extension installed inside Antigravity. Continue speaks Ollama → bridge handles the rest. Activated via ⌘L.

Cursor ⚠ Same as Antigravity — use Continue.dev

Cursor's Cmd+K and chat route through cursor.sh cloud. The "OpenAI base URL" override field that previously existed has been disabled in current builds. Continue.dev v2.0.0 extension installed inside Cursor for the same workaround.

VS Code ✓ via Continue or Cline extension

Install Continue.dev or Cline from the marketplace. Same ~/.continue/config.yaml works everywhere.

Any tool with OPENAI_BASE_URL or Ollama URL ✓ Works

JetBrains AI Assistant, Helix, Neovim plugins, CLI tools — point them at localhost:11434 (Ollama) or http://localhost:11435/v1 (OpenAI). Bridge handles them all.

11 Install from scratch (for your team)

Each teammate runs this once on their own Mac. Total time: under 2 minutes.

📋
Prerequisites: Node.js (brew install node) and hai already configured for Claude Code in Terminal (hai configure claude-code).

2
Run the installer

cd ~/Downloads/hyperspace-bridge
./install.sh

Free tier gives 1,000 searches/month — enough for daily dev use. Without a key, Claude will say "I can't search the web" when asked about live data.

Skipped it? Re-run ./install.sh at any time to add the key. It will detect the missing key and prompt for it.
4
For Zed users — open Assistant and use Claude

Bridge auto-registers in ~/.config/zed/settings.json if you copy the template entry. Restart Zed → ⌘? → pick "Claude (via Hyperspace)".

# Cursor:
"/Applications/Cursor.app/Contents/Resources/app/bin/cursor" \
  --install-extension Continue.continue
6
Create Continue's config — same for every IDE

Save to ~/.continue/config.yaml:

name: Hyperspace Bridge
version: 1.1.0
schema: v1

models:
  - name: Claude Sonnet (Hyperspace)
    provider: ollama
    model: claude-sonnet-latest
    apiBase: http://localhost:11434
    roles: [chat, edit, apply]
    defaultCompletionOptions:
      contextLength: 200000
      maxTokens: 8192

  - name: Claude Haiku (Hyperspace)
    provider: ollama
    model: claude-haiku-latest
    apiBase: http://localhost:11434
    roles: [chat, autocomplete]

  - name: Claude Opus (Hyperspace)
    provider: ollama
    model: claude-opus-latest
    apiBase: http://localhost:11434
    roles: [chat, edit]

Restart the IDE and chat

Open the IDE → Continue icon appears in the sidebar (or press ⌘L) → pick the Claude model → type. Real Claude through Hyperspace, no internet config, no API keys, no cloud bills.

14 Security model

PropertyDetail
Network exposure127.0.0.1 only — bridge binds explicitly to loopback, never 0.0.0.0. Verified by attempting connection from LAN IP 192.168.50.254 → refused.
API key handlingBridge never stores keys. Reads token at startup from ~/.claude/settings.json (same place hai configure claude-code wrote it). Forwarded as x-api-key header on each request.
LoggingBridge logs only method, path, status, and latency to ~/Library/Logs/hyperspace-bridge.log. No request or response bodies are logged.
Caching / persistenceNone — bridge is a transparent proxy. Every request hits Hai fresh.
DependenciesZero npm packages — uses Node's built-in http, fs, path, os only. No supply-chain surface.
Process isolationRuns as your user via launchd. No root, no sudo, no system-wide install.

15 Troubleshooting

The bridge isn't responding

# Check launchd status:
launchctl list | grep hyperspace

# View live log:
tail -f ~/Library/Logs/hyperspace-bridge.log

# Restart manually:
launchctl bootout gui/$(id -u)/com.hyperspace.bridge
launchctl bootstrap gui/$(id -u) ~/Library/LaunchAgents/com.hyperspace.bridge.plist

Continue extension doesn't appear in the IDE sidebar

Fully quit the IDE (⌘Q, not just close window) and relaunch — Continue activates on onStartupFinished. If still missing, verify with:

"/Applications/Antigravity IDE.app/Contents/Resources/app/bin/antigravity-ide" \
  --list-extensions | grep continue

Hai API key not detected

Run hai configure claude-code first — that writes the token into ~/.claude/settings.json where the bridge looks for it. Or set HAI_API_KEY=<your-token> in the launchd plist's EnvironmentVariables.

Wrong port — Hai isn't on :6655

Override at install time:

# Run bridge manually with custom Hai URL:
HYPERSPACE_URL=http://localhost:YOUR_PORT/anthropic node ~/.hyperspace-bridge/hyperspace-bridge.js

Clean reinstall

cd ~/Downloads/hyperspace-bridge
./uninstall.sh
./install.sh
